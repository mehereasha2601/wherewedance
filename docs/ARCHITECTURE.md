# Architecture

This document describes how WhereWeDance is wired together. It is aimed
at engineers picking up the codebase for the first time.

## High-level shape

```text
 ┌─────────────────────────────────────────────────────────────┐
 │ TanStack Start (Vite + React 19, SSR-capable)               │
 │                                                             │
 │  src/routes/*       file-based routes (URL → page)          │
 │     │                                                       │
 │     ▼                                                       │
 │  src/components/pages/*   one component per route           │
 │     │                                                       │
 │     ▼                                                       │
 │  src/components/wwd/*     shared building blocks            │
 │     │                                                       │
 │     ▼                                                       │
 │  src/components/ui/*      Radix/shadcn primitives           │
 │                                                             │
 │  src/data/mock.ts         typed in-memory data (no I/O)     │
 │  src/lib/event-dates.ts   Boston-tz date helpers            │
 │  src/styles.css           Tailwind v4 tokens (oklch)        │
 └─────────────────────────────────────────────────────────────┘
```

There is **no backend**. Every list, filter, and detail page reads
from `src/data/mock.ts`. Forms (Contact, Safety, etc.) are visually
complete but `onSubmit` only calls `alert()` / preventDefault — nothing
is sent anywhere.

## Routing

- Filenames in `src/routes/` map to URLs via TanStack's dot-separated
  convention (`events.$id.tsx` → `/events/$id`, `organizers.index.tsx`
  → `/organizers`).
- Every route file owns its `head()` metadata (title, description,
  og:title, og:description). The shared shell lives in
  `src/routes/__root.tsx`.
- Page components are kept in `src/components/pages/` and imported by
  the route file. Route files stay thin (metadata + component import).
- `src/routeTree.gen.ts` is auto-generated by the TanStack Router
  Vite plugin — never edit it by hand.

## Data layer (`src/data/mock.ts`)

`mock.ts` is the single source of truth for everything user-visible.
It defines TypeScript types (`Organizer`, `Event`, `Resource`, `Buddy`,
`Value`, `AskPrompt`) and exports typed arrays plus small selector
helpers:

- `eventById(id)` / `eventBySlug(slug)`
- `organizerById(id)` / `organizerBySlug(slug)`
- `resourceById(id)`
- `getThisWeekEvents(today?)` — Mon–Sun window in Boston tz
- `tonightEvents(today?)` / `isEventTonight(event, today?)`
- `isPastOneOff(event, today?)` — used by Ask answers and listings to
  filter out one-offs after their date
- `catalogDateLabel(event)` / `getAskDateLabel(event, today?)`
- `mapUrlForEvent(event)` — null when the venue address is too vague
  (e.g. `TBA`, `Location TBA`) to map safely
- `logisticsSummary(event)`, `eventsByDay()`

Adding or editing data is a documentation task as much as a code task —
see [`CONTENT.md`](CONTENT.md) for the editorial rules.

### Coming Soon convention

Any prototype-only page (Ask, Buddies, Contact, Safety, Organizer
Dashboard) renders a single `<ComingSoonBanner>` near the top. The
banner has one consistent pink/magenta visual treatment regardless of
the `variant` prop. Don't build bespoke "this is a prototype" cards
inline anymore — extend `ComingSoonBanner` instead.

### Link safety

`ResourceCard` and `AskAnswer` both follow the same rules:

- An href starting with `/` is routed through TanStack `<Link>`
  **only** if it appears in the `KNOWN_INTERNAL_ROUTES` allowlist
  (`/values`, `/beginner-guide`, `/resources`, `/contact`,
  `/organizers`, `/safety`). Otherwise it renders as plain text so
  the build stays type-safe and no dead links ship.
- An href starting with `http(s)` opens in a new tab with
  `target="_blank" rel="noopener noreferrer"`.

## Styling

- Tailwind CSS v4 is wired via `@tailwindcss/vite`. The token layer
  lives in `src/styles.css` using `@import "tailwindcss"` and
  `@theme { ... }` blocks.
- Brand palette is defined in `oklch` so it stays perceptually even
  across light/dark contexts. Semantic names (`paper`, `ink`,
  `terracotta`, `mango`, `oxblood`, `magenta`) are used everywhere —
  do not introduce ad-hoc hex values in components.
- Typography: Fraunces (display, italic) + Instrument Sans (body) via
  `@fontsource-variable/*`.

## SSR / runtime

- TanStack Start renders routes on the server when published, and
  hydrates on the client.
- There are no server functions or `/api/*` route handlers in use
  today. The app is effectively a static React SPA with SSR enabled
  for SEO / share-image friendliness.
- No environment variables are required at runtime.

## Adding a new page

1. Create the page component in `src/components/pages/MyPage.tsx`.
2. Create the route file in `src/routes/my-page.tsx` (or
   `parent.child.tsx` for nested paths). Set route-specific `head()`
   metadata.
3. Add a nav entry in `bottom-nav.tsx` or `MorePage.tsx` if the page
   should be reachable from the chrome.
4. If the page is a prototype, use `<ComingSoonBanner>` and avoid any
   wording that implies real submissions, AI, messaging, or auth.
